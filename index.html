An offline-first approach to client- server architecture that will make your development Rock

1Make more by doing less

Reflex saves you huge amount of work by providing a ready to use interface.

Forget about creating lots of queries to receive a data from server. Reflex creates a snapshot of all necessary data on a client side, so all you need for productive work is already here.

2When everything works offline

Practically any application can be made offline first using Reflex methodology.

Due to the multiversion nature of data in Reflex you can safely delegate the resolution of conflicts to users and keep all of the further changes fresh.

3And everyone is invited

Reflex was built technology-agnostic, meaning that it is friendly to any language, platform or database.

The original implementation is created for SQL and fully compatible with ACID, so it is easy and comfortable to integrate Reflex with all types of databases.

What’s inside

Client-side storage

Applications built with Reflex are using the data stored on the client side (for web-applications they use technologies like IndexedDB / WebSQL).

Providing the local storage of all necessary data, Reflex makes it possible to fully interact with the application even if the connection with server breaks.

This also allows to reduce the volume of data flowing inside the network. The main part of data is sent to the client with the very first launch of application, and subsequently the new data are being kept fresh.

Multiversion data

Multiversion data is the heart of Reflex. Having the ability to save all versions of one piece of data simultaneously, you can safely edit and save data being offline.

Any of your edits won’t delete the previous changes made by other user when they are unavailable at the moment because of a broken connection.

Reflex will save all versions of data, whereupon it will be possible to delegate the resolution of conflicts to the application or its users. For such resolution you will only have to provide a user with simple UI, allowing him or her to choose a right version of data independently.

Interactivity

The data stored on the client side are always being kept fresh. When needed, Reflex automatically updates new data in real time using reverse-HTTP technologies.

At the moment when other user saves new changes asking a server to do that, every connected client is being notified about these changes

Such approach to working with data allows to avoid the necessity of creating queries. It reduces the amount of code required to build an application.

Partial database replica based on permissions

Any piece of data is created by Reflex with a reference to the permission. Due to the system of permissions user receives only necessary data while other (unavailable) data is being unable to view.

This approach not only improves the security of the developed applications, but also guarantees the distinction of data to users.

It also allows you to control the amount of data being sent to the client. Having a list of authorizations in accordance with the application’s structure, adding or removing user permissions respectively allow to download or delete the necessary data.

Insert-only changes and full history database

All changes on the database side are implemented by inserting. Even removing does not lead to the physical destruction of the data saved previously.

This makes it possible to store all the history of changes and analyze the state of the system at any time in the past when the communication with server becomes available.

All stored data are associated with the user who made those changes, thus this approach also prevents the need for logging changes.

Single endpoint for all data

Reflex uses only one "entry point" (endpoint) for data exchange, that in comparison with typical applications using dozens, hundreds and thousands of endpoints, can greatly simplify the application and increase its reliability.

It is much easier to ensure high-quality, fast and reliable work of single point of entry than to provide the same guarantees to thousands endpoints of common application.

Thus, the exchange of data covered by Reflex, occurs quickly, reliably and securely.

Integration with third party technologies

Reflex is a methodology that has been constructed to be open-minded for third party technologies, e.g. Angular, React, Ember, Knockout, and many more.

This allows to organize the interactive flow between data model and visual representation (UI) for web-applications.

Technology-agnostic

Reflex can work in conjunction with the absolute majority of the existing databases that support a minimal set of operations. It can be as Industry Standard SQL (such as Microsoft SQL Server, Oracle SQL, MySQL, Postgress SQL, et al.), as well as NoSQL databases (MongoDB, RavenDB, Redis, Cassandra, and others.).

To implement the principle of Reflex it is enough for a database to be able to store and query data using a composite logical conditions `AND` and `OR`.

In addition, Reflex is available for your use by any client (web and mobile), any backend and any supporting technology.

Reflex на стороне клиента реализует паттерн репозиторий, иными словами хранилище данных. Что бы создать репозиторий необходимо определить схему данных - обыкновенный JSON. Давайте определим две сущности “City” и “Country”

Это типичная структура данных, где страна и город имеют атомарное поле “name”, в свою очередь City в поле “country” ссылается на страну “Country”. На основании этих данных клиентская библиотека будет способна понять то как должны выглядеть данные скоторыми она будет работать.

После того как схема готова можно создать репозиторий

Но этот репозиторий еще пока пуст, давайте создадим данные для этого репозитория:

Данные изображенные на этом рисунке в таком виде при использовании Reflex не придется создавать в ручную обычно такие данные будут передаваться сервером на клиент автоматически но для явности давайте рассмотрим из каких элементов состоят эти данные.

  1. Первое на что стоит обратить внимание что переменная в которую ложится этот массив называется diff. Причиной тому есть специальный формат обмена данными между репозиториями который был назван diff по аналогии c diff’ом репозитория GIT этот формат имеет такую же природу как и обычный diff и при его помощи реализуются операции слияния, пересечения, суммы и минуса множеств данных о которых более детальный разговор пойдет далее
  2. Второе на что стоит обратить внимание что diff это массив, важно осознавать что от порядка находящемся в этом массиве данных в особых случаях может меняться результат операций над множествами данных, о чем более детально будет написано позже
  3. Элементы массива назвем Record’ами, это сервисные сущности используемые для переноса данных между репозиториями либо базой
  4. Поле prv - это ID предыдущей версии текущего данного, так как это первая версия предыдущей версии еще не существует и по этому это поле равно null
  5. Поле rid - расшифровывается как Reference ID или просто ID сущности, для клиентской библиотеки Reflex - rid может быть любой строкой
  6. Поле vid - расшифровывается как Version ID, для клиентской библиотеки Reflex - rid может быть так-же любой строкой
  7. Поле pid - расшифровывается как Permission ID, это разрешение при помощи которого сервер понимает можно ли текущему пользователю отправлять текущее данное или нет, более детально вопросы сервера и разрешений мы будем обсуждать позже
  8. Поле proto - содержит контейнер с данными в разобранной форме, что означает что ссылки на сущности в этой форме представлены строками
    1. Обратите так-же внимание что для первых двух record’ов поле “proto.country” ссылается на rid соответствующей сущности
  9. Поле table - содержит информацию о том к какой таблице данных относится текущая сущность
